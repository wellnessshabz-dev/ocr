# Auth.md — MCP Night 4: Agentic Registration and the Future of Software

> Reasoned analysis of Michael Grinich's (CEO WorkOS) talk at MCP Night 4, announcing Auth.md — an open spec for agentic registration, with a live demo by Madison (WorkOS engineer).

## The Evolution of Agentic Coding

| Era | Pattern | Human Role |
|-----|---------|------------|
| Year 1 (autocomplete) | Prompt → code snippet | Hand-review every line |
| Year 2 (one-shot features) | Prompt → full feature | Review entire feature |
| Year 3 (long-running agents) | Decompose → agent runs for minutes/hours | Review intent + output |
| **Now** (parallel agents) | Multiple agents collaborate, pick stack | Principal engineer review |

> "Software engineering has changed more in the last four months than over the last forty years."

## The Harness Thesis

The model is not the product. The **harness** is:

1. **Runtime** — Secure execution, permissions, isolation, scoped credentials
2. **Tools** — GitHub, CI, observability, databases, feature flags, issue trackers
3. **Context** — Codebase, PRDs, Notion/Confluence, team conversations, call transcripts
4. **Feedback loops** — Build → test → review → verify
5. **Human review** — Moving up the stack from each line to design decisions

> "Every time models change, we have to rebuild the harnesses. New engine → new chassis."

## Agents as Consumers

**Core insight:** Agents are the new users. Products must be accessible to agents or they die.

- If your product can't connect to an agent → 1,000x less likely to be used
- Agents pick stacks (database, hosting, auth, email, observability)
- Services with better agent-native experiences get selected by agents
- Agent users may be **more valuable** than human users (spawn faster, never sleep, scale infinitely)

> "Make something agents want" — revision of YC's "make something people want"

## The Registration Problem

Decades of web security built to **block** automated behavior. Now we need the opposite.

| Human Registration | Agent Registration |
|-------------------|-------------------|
| Reads landing page | Automated discovery |
| Fills out forms | Needs capability declaration |
| Solves CAPTCHA | Can't solve CAPTCHAs |
| Verifies email | Might not have an email |
| Human-in-the-loop OAuth | Needs automated auth |

MCP is not enough — its auth layer requires human consent through OAuth flow. Agents need something automated.

## Auth.md — Open Spec for Agentic Registration

**Not a standard.** "Just a set of ideas and principles" — an open spec for agent-native signup.

### What Auth.md Answers

1. **What does the service do?** — Capability declaration
2. **How does an agent register?** — Signup protocol
3. **What identity proofs are accepted?** — Trust relationships
4. **What offloads are supported?** — Authentication methods
5. **What scopes/entitlements exist?** — Permissions
6. **What free tier constraints apply?** — Gated experience
7. **How does a human claim the account?** — Organization transfer

### The Flow

```
Agent (harness) → discovers auth.md on service → sends IDJAG → receives access token → uses API/MCP
```

### IDJAG (Identity Assertion JWT Authorization Grant)

A JWT containing:
- `email_verified` — Trust anchor for identity
- `issuer` — The agent/identity provider (e.g., Acme)
- `audience` — The target service (e.g., Cloudflare)

Trust relationship: Service trusts the IDJAG issuer's email verification. No human-in-the-loop OAuth needed.

### Live Demo (Madison, WorkOS Engineer)

1. Agent (Pi harness) working on a Svelte personal website
2. Needs to deploy — discovers Cloudflare supports agent auth via `cloudflare.com/auth.md`
3. Agent sends IDJAG signed by Acme identity (`madison@workos.bot`)
4. Cloudflare trusts Acme's email verification → returns API key + account ID
5. Agent deploys site using Wrangler CLI
6. Same flow for Firecrawl: discovers → IDJAG → API key → scrapes web design museum
7. Agent reskins the site (pixel art theme) and deploys

No terminal exit, no browser, no manual signup forms.
